# 5 - The Process Definition

This Process is made of building blocks covering various aspects of the software, e.g., It defines the Database through Class Diagrams; The Menu, and from the menu some CRUD code for simple data access; Some Intermediate Level Queries, from where It’s possible to ask question to the database making List and Form Activities; And It also defines a Vocabulary which is applied to State Machine Diagrams that are converted directly in to code; The Classes not only represent Tables but also and sets of variables to be used by the Behavior Diagrams, depending on Some specific Stereotypes which I will point to them. Another important building block covered by this Framework is the Automatic generation of the Database Synchronization code. This process has the follow steps:

  • Creating the structure model:

    • Drawing the package diagram

      • Each package is going to originate a [module] of your system

    • Drawing a class diagram for each package

  • Fixing the types model:

    • You may need to set your own custom types for the business you’re modeling

  • Creating the behavior model:

    • Drawing the initial behavior diagram

      • It’s not strictly necessary to have a complete model, but it’s necessary to create a file for it

      • At this first version you can start building your menu, and maybe starting some [state-machines]

  • Mixing the structure model into the behavior model (using a model to model updating transformation):

    • This will generate a "mvc" package composed of:

      • The models package

        • Responsible for the direct database generation

      • The views package

        • It represents all the screens and parameters

      • The controllers package

        • It may have the same classes of the model class, but the only content of the classes is a example function which you need to update afterwards. Please checkout [the queries profile]

  • Adjusting the Behavior model:

    • A little model to text transformation has been made to check for problems in the model:

      • It shows properties which have no types

      • It shows foreign keys properties pointing to it’s owner class (maybe it can be ignored)

      • It also shows views which repeated property names, It’s an error which is still not fixed during the "mixing transformation" a property with the same name and a different type is just added

  • Adding custom queries

    • The queries are going to be used afterwards to give some "direction" to the behavior diagrams. They are (together with the web services) the "question points" from where you can ask things about the "products" or even about your "clients" in order to take your business decisions

  • Fixing the extended state machine diagrams (which in our case don’t represent states)

    • It will probably happen at the same time you build the custom queries

    • At the current version the transformation code for state machines is unstable (you may pass forwards through this)

  • Running various model to text transformations:

    Once it's a big project, I would like to make small deliveries, from the Database to the Sync code to the Strings (translations) and Pages and at the end giving you the possibility of defining the behavior of your applications. Testing is a really complicated part of this project I hope it will be more clear how to deal with it in the future. But now it's just trying and error, building and running right away.

    • The "Model-to-Android-Project" Transformation

      • It’s not recommended to use right now (better making all the applications on android studio or eclipse)

    • The "Model-to-Database" Transformation

      • I’m sorry but at the current time It’s mixed with the content provider transformation. It should probably change in the future

    • The "Model-to-Database-Synchronization" Transformation

      • It will generate some PHP code (at the moment we don’t generate anything specific for any famous PHP framework)

      • It will generate the Android database synchronization classes

        • There is an special Transformation build in CoffeScript responsible to sort the tables according to the foreign key "pointers", whatever is pointed by a foreign key always come first.

    • The "Model-to-Menu" Transformation

      • It was working, but after the code refactoring. I may say that it’s not working. But it’s a simple transformation

      • From this transformation it will also be generated "Activities" (or pages) for the menu items. If it’s a crud page pointing to a table, it will be very straightforward, because a CRUD is a well known Design Pattern
