# 2 - Related Works

PRADA (2012) Proposes the use of Sequence Diagrams in order to generate android applications. According to PRADA (2012) on they approach is not appropriate for variable attribution and simple math operations. KRAMER (2011) Instead uses Activity Diagrams on a very elegant way, including, extending the UML Language with some Stereotypes in order to solve the problem. According to KRAEMER (2011) the code generation process can result in building Android Applications 4 times faster. Of course considering the right-to-use transformation process and not the process of making the transformation itself. FREITAS (2016) Proposes a three phases transformation process. The Model Generation, by using the JBModel tool, the Transformation of the model into annotated Java classes and an additional code generation by using a tool called JBusiness. But according to FREITAS (2016) the user don’t need to focus on programming at all. The current work is similar to the above works but uses State Machine diagrams to represent everything on the System. But the main difference is that The State Machine diagram is not used as a State Machine but as an empty space from where it’s possible to fill the meaning by using Stereotypes. GOAER (2014) presents a work which seems to be similar at the first sight, but his/her proposal refers to executing the model directly in the android device. Also none of them seem to be concerned about Structural Diagrams. This work also differs from the previous, being shown, because it aims to make the Best effort to Ignore the fact that it’s an Android or Web Application providing common Building Blocks. And it’s also different because It Actually can’t reach the complete yet.
