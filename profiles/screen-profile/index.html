# The Screen Profile

This profile is intended to help making many different kinds of screens by providing the elements which are supposed to be present on each screen. In order to make form fields we are making the use of Classes, but by using uml::Class we don't represent buttons properly so the missing buttons are represented by uml:States on a (special made) State Machine Diagram.

# Stereotypes

# MainMenu

The MainMenu indicates that this state machine is the first one to be converted in to code. Each single MenuItem present on it will probably generate activities.

# Menu Items and Sub-Items

Each single state on a MainMenu State Machine will represent the MenuItems and each "grandchildren or "grand-grandchildren" are supposed to represent the "Sub-Items" or "Sub-Sub-Items" respectively.

# Stereotypes representing Pages (which will originate Activities or Controls)

# Page

The Page means that from that the Menu Item (represented by a state) will become an Activity (or Page), but no special code will be generated. The behavior is determined by it's Sub-Items (In general for direct Database Access).

# CrudPage

The CrudPage means that the current Item (represented by a state) will generate an activity with all the operations for direct access to a Database, including maybe search by name (if this field is present), List, Insert Update and Delete.

  • A View Must be provided ** In this case the View can be a class representing a Single Table or a View Class

# CustomPage

Means that the current item represents a sequence of actions, which means Generating an activity where it's behavior is defined by another state machine (until the current time it wasn't defined yet as a pointer to a state machine)

# Stereotypes representing different kinds of Regions on a Page

# FormRegion

This is the most general of all Stereotypes Representing Region

  • A View Must Be provided to this form in order to generate the Inputs
  • The Buttons will be represented by sub-states and they will generate a Sequence of actions represented by the outgoing transitions between the "Button" and another states

# InsertRegion

This represents booth the inserting and updating Forms (or Activity)

  • A View Must be provided

# ListRegion

This means that on that the given State where its applied represents a list of "Entities". A generic template will be used to generate the list.

  • A View Must be provided

# Stereotypes representing form "Controls"

# Barcode

The Barcode is intended to be applied to a uml::Property It's used when we want to use a previous made Activity to read data from a Barcode Picture.

# DateTime

It menas that the Data which will supply the current Field (Represented by a uml::Property ) comes from some sort of DateTime Control (Or Activity), and the value provided by the control represents the Current Date and Time

# Date

Same as the previous, but with the difference that only the Date should be provided by this control

# Time

Same as the previous, but with the difference that only the Time should be provided by this control

# HasPicture

This indicates that a given "Form" will take pictures of a given "Entity" It will call the proper "Picture Activity" (or Control) to add the Picture of the "Entity" which is gonna be represented by a uml::Class which represents a "Table", it's not applied to a "Field" ( uml::Property ) because the way it's implemented it's supposed to use the Id of the Class as the foreign key and Picture name (in case the picture is saved as a file).

# Button

It represents a Button, and it's default event is Button-Click. The Button-Click stereotype shall be avoided on a Button
